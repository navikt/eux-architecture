# SED Categorization in RINA / eux-rina-api

## 1. The Two Conceptual Levels

In RINA, an action is either **case-level** or **document-level**. The application surfaces this as two separate fields on `RinasakOversiktV5`:

| Field                       | Meaning                                                                                   | Source in code                                                                                            |
| --------------------------- | ----------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| `sakshandlinger`            | Things you can do **to the case** (start a new SED, close, change participants, …)        | `RinasakOversiktV5Service.extractSakshandlinger()` — filters `actions` where `isDocumentRelated == false` |
| `sedHandlinger`             | Things you can do **to a specific SED** (Read, Update, X008-Reject, ReadParticipants, …)  | `extractSedhandlinger()` — actions where `documentId` or `parentDocumentId` matches the SED               |
| `svarsedType` / `svarsedId` | The reply-SED slot for a given SED (a "Create" action with `parentDocumentId` → that SED) | `extractSvarSedInfo()`                                                                                    |

A "sakshandling" template is **either** a SED type (e.g. `H001`, `X005`) **or** a non-SED operation (`Close_Case`, `SelectParticipant`/`singleParticipant`, `Create_Participants`/`multipleParticipants`).

## 2. SED Roles Inside a BUC (from RINA-6.3 `po_*.xml` configs)

Every BUC declares its SEDs in five roles in the config-file comments:

| Role          | Meaning                                                             | What it produces in the API                                                 |
| ------------- | ------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| **STARTERS**  | The SED that **opens** a case of this BUC. The very first document. | Appears as a `sakshandling` ("Create_Case" + this template) on a fresh case |
| **POCREATED** | SEDs the **Process Owner** can create after start                   | Becomes a `sakshandling` on the case while PO owns it                       |
| **CPCREATED** | SEDs a **CounterParty** can create — typically replies              | Becomes a **`svarsed`** on the parent SED (or a new sakshandling)           |
| **HSUBS**     | Horizontal sub-processes (`H1` = info request, `H2` reminder, etc.) | Available as sakshandlinger across many BUCs                                |
| **XADMINS**   | Administrative SEDs available throughout the BUC                    | Always available as sakshandlinger / sedHandlinger                          |

### Schema-defined SEDs not referenced in active 4.3 BUC configs
These exist in the CDM 4.3 catalogue (`eessi-rina-bmp/tools/xmlGenerator/SEDs/`) but are not part of any `STARTERS / POCREATED / CPCREATED` list in the 4.3 BUC `po_*.xml` files. They are most likely **legacy / 4.0-4.2-only**, **engine-internal**, or **planned/reserved**:

| Family                         | Unused SEDs                                                                           | Likely role                                                                                         |
| ------------------------------ | ------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **A** (Legislation Applicable) | A001, A002, A011                                                                      | Older / pre-4.3 LA flow variants                                                                    |
| **DA** (Accident at Work)      | DA010, DA011, DA012, DA012C, DA012R, DA014, DA015, DA016, DA016A, DA017, DA018, DA019 | Older AW BUCs (DA001–DA019 belong to legacy `aw_buc_05`)                                            |
| **F** (Family)                 | F004, F005, F018, F019, F020, F021, F022, F023, F026, F027                            | Used by `fb_buc_01` / `fb_buc_03` (empty in 4.3) — older family-benefit flows                       |
| **S** (Sickness)               | S009, S010, S011, S012, S014, S015, S035, S037, S071, S075, S076, S080–S092           | Used by `s_buc_02, _04, _06–_24` (empty in 4.3) — many sickness BUCs are deprecated/dormant         |
| **U** (Unemployment)           | U004_IT, U007, U008, U009, U010, U011, U012, U013, U014, U015, U016                   | Used by `ub_buc_02 / ub_buc_04` (empty in 4.3); U004_IT is an Italian country-variant of U004       |
| **SYN**                        | SYN001, SYN002, SYN003, SYN004, SYN005                                                | **Engine-internal synchronization** SEDs (used by RINA infrastructure, not exposed as user actions) |
| **X**                          | X010, X011, X012, X013, X050, X100                                                    | X050 = business exception receive-only; X100 = error/system; X010-X013 reserved/admin extensions    |

**Why these matter for eux-rina-api:** if a case from an older BUC (still active in production with CDM ≤4.2) is fetched, you may see SEDs like S010, U010, F018, DA014, etc. The categorization logic in `RinasakOversiktV5Service` doesn't care — it just uses whatever `actions` RINA returns — so they'll be classified correctly as `sakshandling`/`sedHandling`/`svarsed` based on the action's `isDocumentRelated`/`parentDocumentId` flags, regardless of CDM version.

## 4. How the App Determines These Categories at Runtime

Code in `RinasakOversiktV5Service`:

```java
// sakshandlinger: case-level actions
actions.stream()
   .filter(a -> !a.isIsDocumentRelated())
   .map(ActionDto::getTemplate)
   // SelectParticipant -> singleParticipant, Create_Participants -> multipleParticipants
```

```java
// svarsed: the reply slot for a SED
actions.stream()
   .filter(a -> sedId.equals(a.getParentDocumentId())
             && "Create".equals(a.getName())
             && sedType.charAt(0) == a.getDocumentType().charAt(0))   // family match
   // F002 received => F003 reply (special rule for fb_buc_04)
```

```java
// sedHandlinger: anything tied to a specific SED
actions.stream()
  .filter(a -> sedId.equals(a.getDocumentId()) || sedId.equals(a.getParentDocumentId()))
  .map(a -> sedId.equals(a.getDocumentId()) ? a.getName() : a.getTemplate())
```

So the categories are **derived dynamically from the BUC's allowed actions** for the current case state; the lists above (from RINA-6.3 BUC XMLs) tell you the *static* catalogue of which SED can play which role in which BUC.

## 5. Quick Mental Model

```
Case (BUC)
 ├── sakshandlinger        ← case-level: STARTERS + POCREATED + HSUBS + XADMINS + Close_Case + (single|multiple)Participant
 └── sedListe[*]
      ├── sedHandlinger    ← per-SED: Read, Update, Delete, Send, X008(Invalidate), ReadParticipants…
      └── svarsedType/Id   ← present iff the BUC defines a "Create" action with parentDocumentId == this SED
                             (the reply belongs to CPCREATED in CDM, e.g. U001→U002, P2000→P3000…)
```
